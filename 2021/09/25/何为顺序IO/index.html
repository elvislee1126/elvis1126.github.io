何为顺序IO

Posted on 2021-09-25 Edited on 2021-12-22

前言

​ levelDB、rocksDB 等键值对数据库在处理客户端传来的 set key、del key 等命令前，先会把此次操作追加写入到 WAL（write ahead log） 日志文件中，使得内存数据丢失时，仍可以通过磁盘中的WAL日志进行恢复。对于键值对数据库这样一种常用作缓存的面对巨量请求的中间件而已，经历这样一次从用户态到内核态到机械层面的磁盘写入，毫无疑问会成为性能的瓶颈。
然而，有一种说法是追加写入日志文件的方式属于磁盘顺序写入，其写入速度堪比内存，这就让我十分困惑：

  • 到底什么是磁盘「顺序写入」？
  • 那「随机写入」又是什么？
  • 磁盘顺序写入为什么「速度堪比内存」？

介绍

虚拟文件系统 - virtual filesystem [switch]

​ VFS 是操作系统内核中的一个软件层，是对具体的各类文件系统的抽象。VFS 对用户空间的进程提供一组文件系统的系统调用：open(2) stat(2) read(2) write(2) chmod(2) 等等。通过 VFS 也实现了多种的文件系统具体实现在内核空间的共存。

目录项 - directory entry

目录项缓存 - directory entry cache (dcache)

​ 目录项缓存，简称 dentry cache，提供了一种高性能的查找机制可以快速的把 open(2) stat(2) read(2) write(2) chmod(2) 等方法的 pathname/filename 参数翻译成一个具体的 dentry。dentry cache 永远只存在于内存中而不会持久化到存储设备上。

​ dentry cache 是你的整个文件空间的一个视图，部分设备可能无法把整个 dentry cache 装进内存中，？VFS 会通过查找 inode 补全 dentry cache。

索引节点 - index node（inode）

​ Inodes 可以是一个常规的文件、目录、有名管道 FIFOs 等。inode 是存在于磁盘上的，当 VFS 需要具体的某些 inodes 时，会把它们从磁盘拷贝到内存中，并且对它们的修改会写回到磁盘中。

​ dentry 和 inode 是 n:1 的关系，每一个 dentry 有一个指针指向一个 inode，一个 inode 可以被多个 dentry 所引用（比如硬连接的场景下）。

​ ？查找一个 inode 要求 VFS 调用父目录的 inode 的 lookup() 方法，这个方法被该 inode 的所属文件系统所装载。当 VFS 有了所需的 dentry（和 inode），便可以实现 open(2) 一个文件，stat(2) 一个文件之类的方法。stat(2) 方法非常简单，VFS 获取 dentry 所指向的 inode，把 inode 中的数据返回到用户空间即可。

The File Object - file 结构体

内核用于管理一个被某个进程所打开的文件

​ file 结构体是内核空间的文件描述符的实现。file 结构体有一个指针指向一个 dentry，并且有一组由 inode 提供的 file operation 方法。因此，进程打开一个文件的过程如下：

  1. 首先会分配一个 file 结构体
  2. 根据 pathname 参数查找从 dentry cahce 中快速查找到 dentry
  3. 有了 dentry 以及 dentry 中指向的 inode 后，便可以初始化 file 结构体
  4. 初始化 file 结构体后，调用由 inode 提供的一组 file operation 方法中的 open() 方法，以让该 inode 所属的具体文件系统实现去做一些相关的操作。
  5. 接着，分配一个用户空间的该进程的文件描述符 fd（就是一个数字）
  6. fd作为 key，file 结构体作为 value，设置到该进程的文件描述表中即可。

可以看到，VFS 向用户空间暴露的就是一个数字文件描述符，简称 fd，

诸如 Reading、Writing、Closing 等方法都是通过 fd 通过进程的文件描述符表查找到对应的 file 结构体调用相应的 file operation 方法组中的方法。
